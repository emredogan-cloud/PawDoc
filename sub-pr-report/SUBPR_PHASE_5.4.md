## 2. CR #11 — Localized Emergency Keywords (the SAFETY GATE)

The same data + the same selector function exist on both runtimes, and the request-time `locale` value drives the lookup on **both** sides.

**Python — `ai-service/app/safety.py`:**
```python
EMERGENCY_KEYWORDS_BY_LOCALE: dict[str, list[str]] = {
  "en": ["not breathing", ..., "compound fracture"],   # 23 unchanged
  "de": ["atmet nicht", "krampfanfall", "verdacht auf vergiftung", ...],
}
SPECIES_EMERGENCY_KEYWORDS_BY_LOCALE: dict[str, dict[str, list[str]]] = {
  "en": {"rabbit": [...], "guinea_pig": [...], "bird": [...], "reptile": [...]},
  "de": {"rabbit": ["frisst nicht", ...], "guinea_pig": [...], "bird": [...], "reptile": [...]},
}
EMERGENCY_KEYWORDS              = EMERGENCY_KEYWORDS_BY_LOCALE["en"]            # alias
SPECIES_EMERGENCY_KEYWORDS      = SPECIES_EMERGENCY_KEYWORDS_BY_LOCALE["en"]    # alias

def _norm_locale(locale):                       # "de-DE" -> "de"; unknown -> "en"
    code = (locale or "en").strip().lower().split("-")[0]
    return code if code in EMERGENCY_KEYWORDS_BY_LOCALE else "en"

def check_emergency_override(text, species=None, locale="en"):
    lkey = _norm_locale(locale)
    for kw in EMERGENCY_KEYWORDS_BY_LOCALE[lkey]:                ...
    for kw in SPECIES_EMERGENCY_KEYWORDS_BY_LOCALE.get(lkey, {}).get(_norm_species(species), ()):  ...
```

**JS mirror — `supabase/functions/_shared/emergency_keywords.mjs`:** the same locale-keyed dicts and a `normLocale` that also falls back to `"en"` for unknown codes; `containsEmergencyKeyword(text, species, locale)` is exported and now takes the third arg.

**Pipeline selection:**
- `AnalyzeRequest.locale: str = "en"` (Pydantic model, default `"en"`).
- The pipeline calls `check_emergency_override(request.text_description, request.pet.species, request.locale)` **before any AI call** — exactly the same order as before, just locale-aware now.

**Edge `/analyze` wires the locale end-to-end:**
1. `select preferred_locale` is added to the user-profile read.
2. `const locale = body?.locale || profile?.preferred_locale || "en"` — the request body may override (web client), otherwise the user's stored preference, otherwise `"en"`.
3. `containsEmergencyKeyword(text_description, pet.species, locale)` — the paywall-bypass also localized so a German emergency text never paywalls.
4. The AI-service POST body now carries `locale`.

**Edge `/analyze-anonymous`** accepts `body.locale` (the German web checker passes `"de"`).

**Proof tests (the safety contract):**
- Pytest — `test_de_global_keyword_fires_under_locale_de`, `test_de_species_keyword_is_species_specific`, `test_locales_do_not_cross_match`, `test_unknown_locale_falls_back_to_english_safe_default`, `test_bcp47_de_de_normalizes_to_de`, `test_de_species_keys_match_en_species_keys` — all green.
- Node — mirror tests on the JS side + `BCP-47 like 'de-DE' normalizes to 'de'` + EN-alias parity test — all green.

**Why this is a safety gate, not a translation niceness:** if `preferred_locale = "de"` and "Krampfanfall" (German for "seizure") didn't trip the override, a German user with a seizing pet would get a normal Tier 2 routing — confidence-gated, latency-introducing, and potentially mis-triaged. The German launch *cannot* happen until those keys exist on both Python and JS — they now do, and tests assert the cross-language locale doesn't false-cross-match.

## 3. How B2B-Lite alters RLS / limits

**RLS — unchanged.** `pets`, `health_journals`, `analyses`, `referrals` etc. all remain own-row by `user_id`. A sitter manages many "clients" under one account, so the sitter's user_id owns every row — the rows simply carry a cosmetic `client_name` label so the sitter can tell them apart. We *deliberately* did not introduce a per-client sub-tenancy model (would have needed a `clients` table + a `client_id` FK on every pet row + new RLS policies + a join in every existing query), which would be a much larger surface area and outside this sub-PR's scope. The label-only approach lets a sitter manage many pet owners without any RLS hole.

**Limits / paywalls — extended uniformly:**

| Gate / location | Before | After |
|-----------------|--------|-------|
| `pet_limits.petLimitFor(status)` | unlimited only for `family` | unlimited for `family` **and `b2b_lite`** (`_unlimitedPetTiers` set) |
| `UserProfile.isPremium` | `{premium, family, trial}` | `{premium, family, trial, b2b_lite}` (`_premiumTiers`) |
| Edge `/analyze` `PREMIUM_STATUSES` | `{premium, family, trial}` | `{premium, family, trial, b2b_lite}` — free-tier counter not incremented |
| `pets_pending_journal` RPC | `subscription_status in ('premium','family','trial')` | `(... , 'b2b_lite')` — sitter pets with the journal opt-in get the weekly narrative |
| `paywall_policy.shouldShowPaywall` | bypasses when `isPremium` | unchanged — picks up B2B-Lite via the `UserProfile.isPremium` change |
| `_shared/revenuecat.mjs` mapping | always returns `premium` on a paid event | reads `event.entitlement_ids` (or `entitlement_id`) — `b2b_lite` → `'b2b_lite'`, `family` → `'family'`, unknown / missing → `'premium'` (preserves existing customers) |

**Schema:**
- `pets.client_name text` (nullable, cosmetic — no CHECK constraint).
- `users.subscription_status` is text; documentation comment updated; `b2b_lite` is one more accepted value with no schema CHECK (same pattern as existing tiers).
- New eligibility migration uses `CREATE OR REPLACE FUNCTION` + the original `REVOKE ALL`/`GRANT EXECUTE TO service_role` lockdown — exactly the same security posture as the 5.3 RPC.

**Tested end-to-end:** the journals pg test now seeds an extra `b2b_lite` sitter G with two clients' pets (Coco / Bunny under different `client_name` values) and asserts `pets_pending_journal('2026-05-25')` returns both — alongside the existing premium / family / trial users. Per-user RLS visibility, lockdowns, and idempotency are still proven by the existing assertions.
