# APPENDIX A — PawDoc Store Review Checklist

_Scope: `feat/legal-portal-integration` (main + legal portal, PR #78 unmerged). Date 2026-07-06. Safety-critical pet-health AI triage app — a false negative is the #1 business risk. Every item below is a store/legal/privacy/monetization gate for that specific product. Format: `- [ ] item — rationale — acceptance`. Finding IDs cite the audit digest._

## Apple App Store

- [ ] Create the missing iOS entitlements file and enable Sign in with Apple + Push Notification capabilities (**APPL-01**) — a signed build has NO entitlements file today, so Sign in with Apple (`auth_controller.dart:31`) and push silently fail under review, and SIWA is mandatory because the app offers third-party/social login — App reviewers will reject the build outright — App builds signed with the distribution profile launch, Apple login completes, and a push token is obtained on a real device.
- [ ] Register the `pawdoc://` URL scheme in `Info.plist` (**REC-02**) — iOS never registers the scheme, so password-reset and every deep link are silently broken on iOS and no report ever tested iOS — tapping a `pawdoc://` reset link on a physical iPhone opens the app to the correct route.
- [ ] Add `ITSAppUsesNonExemptEncryption` to `Info.plist` (**APPL-04**) — its absence stalls automated TestFlight/submission processing and forces a manual export-compliance answer each upload — key present (set `false` if only HTTPS/standard crypto is used) and TestFlight processing completes without an encryption prompt.
- [ ] Replace the placeholder App Review demo account with a real, provisioned login and confirm live IAP offerings (**APPL-03**) — the app is auth-gated, so a placeholder account blocks review, and the paywall dead-ends on "coming soon" if RevenueCat offerings aren't live at review time — reviewer can sign in with the supplied credentials and see a purchasable, priced subscription (not "coming soon").
- [ ] Point App Store Connect Support URL, Privacy Policy URL, and review notes at the live legal portal, not `pawdoc.app` (**APPL-02**, **REC-04**) — metadata still references the dead `pawdoc.app` domain while the working portal is a CloudFront URL — every submitted URL returns HTTP 200 and resolves to the correct live page.
- [ ] Confirm no photo-library / microphone purpose strings are declared (keep the surface minimal) (**store-apple**) — the app uses only the `camera` plugin with `enableAudio:false` (`camera_screen.dart:42`, `video_capture_screen.dart:44`) and has no gallery picker, so declaring `NSPhotoLibrary*`/`NSMicrophone*` would invite a "why do you need this" rejection — `Info.plist` contains camera + location strings only; no unused media/mic strings.
- [ ] Verify the medical/AI disclaimer is present and unstrippable in the reviewed build (**LEG portal / disclaimer**) — Apple scrutinizes health apps for diagnosis overclaims; the disclaimer is server-forced (`disclaimer_required=True` in `pipeline.py`) and UI-gated on both result screens — `verify-disclaimers.sh` passes green and the disclaimer renders on-device on both EMERGENCY and non-emergency results.

## Google Play

- [ ] Re-sign the release AAB with a real upload/release keystore — remove the debug key (**SEC-01 / INF-01 / PLAY-01 / REC-01, CRITICAL**) — the release AAB is signed with the well-known debug keystore, which Play rejects on upload and which breaks update integrity — a `bundletool`/`apksigner` verification shows a non-debug signing certificate and the AAB uploads to the Play Console without a signature rejection.
- [ ] Confirm `targetSdk` meets the Play API floor (**store-google**) — `targetSdk` resolves to 36 via Flutter 3.41.9 default, exceeding Play's API-35 requirement — merged manifest shows `targetSdkVersion 36`.
- [ ] Guard OneSignal init so builds without `ONESIGNAL_APP_ID` don't crash on exit (**QA-03**) — the crash-on-exit is unfixed in code and only mitigated by config, risking a Play "stability" flag / bad-behavior policy hit — a release build with no `ONESIGNAL_APP_ID` backgrounds and exits cleanly with no crash in logcat.
- [ ] Downgrade `ACCESS_FINE_LOCATION` to `ACCESS_COARSE_LOCATION` (**PLAY-03**) — FINE is over-declared for a feature that only uses medium accuracy; over-declared location triggers Play permissions review and a Data Safety mismatch — merged manifest declares coarse location only and the vet-finder still functions.
- [ ] Provide a working web account-deletion URL for the Play data-deletion requirement (**PLAY-04 / LEG-02**) — the Play data-deletion URL currently routes to an unprovisioned `pawdoc.app` mailbox, so deletion requests are undeliverable (Play requires a functioning off-app deletion path) — the submitted URL loads a live deletion-request page/form whose contact address has a valid MX record and receives mail.
- [ ] Keep store-listing copy free of "diagnose / treat / cure" claims and retain the emergency-never-paywalled statement (**store-google**) — Play's health-content policy penalizes medical overclaims; current copy is policy-clean — final listing text contains the medical disclaimer and no prohibited claims.

## Required Screenshots

- [ ] Capture all phone screenshots in DARK mode only until UX-01 is fixed (**UX-01**) — in LIGHT mode 13 screens force a dark background under theme-default (near-black) text, producing near-invisible safety guidance; dark portrait is ~90% done and renders cleanly (see `runtime/legal_validation/*.png`) — every submitted screenshot is dark-mode and passes a legibility check on result/safety text.
- [ ] Provide the required device-size sets for both stores — iPhone 6.7"/6.5" (and 5.5" if targeted) plus iPad if iPad is supported; Play phone + 7"/10" tablet if tablet-listed (**store-apple / store-google**) — each store enforces minimum screenshot counts per declared device class — the correct count and resolution per class is uploaded and accepted without a metadata error.
- [ ] Do NOT ship landscape/tablet screenshots that expose edge-to-edge content sprawl (**UX-02**) — `maxContentWidth (480)` is applied on only 2 of ~15 screens, so landscape/tablet captures (e.g. `02_app_launch.png`) show content sprawling full-width — no submitted screenshot shows unbounded full-width layout, or UX-02 is fixed first.
- [ ] Include a screenshot showing the medical/AI disclaimer and the EMERGENCY result path (**disclaimer / emergency**) — reviewers of a health app look for honest framing; showing the disclaimer and the red emergency screen preempts overclaim concerns — at least one screenshot shows the disclaimer and one shows the EMERGENCY result with its vet CTA.
- [ ] Ensure no screenshot text overclaims triage as diagnosis and none advertises `pawdoc.app` (**PRD-02 / UX-04**) — one onboarding headline implies the app replaces vet judgment and share text advertises the dead domain — captioned/overlay text is scrubbed of diagnosis claims and dead-domain URLs.

## Required Metadata

- [ ] Replace every `pawdoc.app` reference in support/marketing/privacy metadata with the live portal URL (**APPL-02 / REC-04 / INF-07**) — metadata still points at the dead domain; the "no-placeholders" CI gate is falsely green while placeholder store URLs remain — a grep of the submitted metadata finds zero `pawdoc.app` occurrences and the CI placeholder gate is red-until-clean.
- [ ] Fix the overclaiming onboarding/value-prop headline in store descriptions (**PRD-02, HIGH**) — the headline implies the app replaces vet judgment, which a health-triage app must never claim and which both stores police — final description frames the app as triage/guidance, explicitly not a diagnosis or vet replacement.
- [ ] Set the correct content rating and age declaration consistent with the 18+ Terms (**LEG-05**) — Terms/Children pages assert 18+ but the app has no in-app age gate, creating a rating/ToS mismatch — store age rating and the in-app posture agree (or the 18+ claim is reconciled).
- [ ] Verify Privacy Policy and Terms URLs in metadata return HTTP 200 on the live host (**LEG portal**) — store review fails on unreachable legal links; the portal's 16 URLs return 200 but metadata may still point elsewhere — each metadata legal URL is fetched and returns 200 to the live page.
- [ ] State the category as Medical/Health-adjacent honestly with disclaimer language in the long description (**health app**) — miscategorization or missing disclaimer invites rejection for a triage product — description includes the "not a substitute for professional veterinary care / call a vet in an emergency" disclaimer.

## Privacy

- [ ] Fill all data-controller identity and EU-representative placeholders in the published privacy/terms/deletion pages (**LEG-01, CRITICAL, blocks both**) — the live documents still carry unfilled controller-identity and EU-rep placeholders, which is a hard privacy-policy-link failure for both stores and a GDPR gap — no placeholder/bracketed text remains on any published legal page and a named controller + EU representative appear.
- [ ] Provision a deliverable DSAR / privacy contact channel (MX record on the contact domain) (**LEG-02, HIGH, blocks Google**) — the sole DSAR/deletion/support address is on `pawdoc.app`, which has no MX record and cannot receive mail — a test email to the published privacy contact is delivered and answerable.
- [ ] Add affirmative Terms/Privacy acceptance at signup (**LEG-03**) — legal links are passive footer buttons with no explicit assent, weakening enforceability and failing consent expectations for a health app collecting sensitive data — signup requires an explicit, logged accept action before an account is created.
- [ ] Resolve divergent duplicate legal docs still marked TEMPLATE with a conflicting DPO contact (**LEG-04**) — `docs/legal/` duplicates carry TEMPLATE warnings and a different DPO contact than the live portal, creating contradictory published privacy terms — a single authoritative set of legal docs exists with one consistent DPO/contact.
- [ ] Disclose runtime third-party network calls (Google Fonts fetch) or eliminate them (**ENG-01 / PERF-03**) — brand fonts are fetched from Google at runtime, an unconsented third-party network call on first launch that must be reflected in the privacy disclosure or removed — fonts are bundled locally (no runtime fetch) OR the Google Fonts data flow is disclosed in the privacy policy.
- [ ] Obtain real attorney sign-off and clear all "counsel-to-confirm" brackets before submission (**legal-compliance**) — several counsel-to-confirm brackets are live and no attorney has signed off, leaving the privacy posture unenforceable — legal docs are attorney-reviewed with no open brackets.

## Subscriptions

- [ ] Keep the auto-renew disclosure plus functional Terms/Privacy/Subscription links on the paywall (**store-apple / store-google**) — Apple 3.1.2 and Play both require an auto-renew/cancellation disclosure with working legal links; these exist at `paywall_screen.dart:220-266` — the paywall shows renewal terms, price/period, and tappable Terms + Privacy links that load live pages.
- [ ] Make Restore Purchases actually work (**SUB-01, HIGH**) — Restore is a silent no-op (no feedback, no navigation, no entitlement refresh), which Apple explicitly requires to function for auto-renewable subscriptions — tapping Restore on a device with a prior purchase re-grants premium and shows confirmation.
- [ ] Add a client-side SDK entitlement fallback so premium isn't 100% webhook-dependent (**SUB-02**) — premium recognition relies entirely on the RevenueCat webhook, so a paid user is blocked until the webhook lands (or forever if misconfigured) — after purchase, the client reads the entitlement from the SDK and unlocks premium even before/without the webhook.
- [ ] Ensure live, priced subscription offerings are configured in both stores before review (**APPL-03 / PRD-05**) — offerings not live cause the paywall to dead-end on "coming soon," which fails review and blocks purchase testing — the paywall renders a real product with price and period pulled from the store.
- [ ] Verify EMERGENCY is never paywalled or quota-blocked, server- and client-side (**monetization-rc, safety-critical**) — the #1 risk is a blocked emergency; logic is enforced at `analyze/index.ts:125-138` and `paywall_policy.dart:28` and unit-tested (8/8 quota-gate cases) — a 0-quota emergency (text and visual) returns a free, uncounted result and never shows the paywall, re-verified on device (**QA-05**).
- [ ] Reconcile the client "free checks left" counter with server bonus credits + monthly reset (**SUB-03**) — the UI ignores referral bonus credits and the monthly reset the server honors, misrepresenting entitlement to the user — the displayed remaining-checks count matches the server's authoritative value including bonuses and resets.
- [ ] Replace the deprecated `purchasePackage` API (**SUB-05**) — using a deprecated purchase API risks future SDK/store breakage — purchases use the current RevenueCat purchase API and complete on device.

## Permissions

- [ ] Keep camera + coarse location as the only runtime permissions; confirm RECORD_AUDIO / READ_MEDIA / storage remain stripped (**store-google**) — the permission diet is disciplined (stripped via `tools:node="remove"`); over-declared permissions trigger Play review and Data Safety mismatches — the merged Android manifest lists only camera + coarse location (no audio/media/storage).
- [ ] Ensure iOS camera and location purpose strings stay specific and justified (**store-apple**) — `NSCameraUsageDescription` (`Info.plist:69`) and location (`Info.plist:71`) are present and specific; vague strings are rejected — both strings name the concrete feature (photo/video triage capture; vet-finder) and no unused permission strings are added.
- [ ] Confirm audio is disabled on video capture so no microphone permission is implied (**store-apple / store-google**) — `enableAudio:false` (`video_capture_screen.dart:44`) keeps mic out of the permission surface; any audio path would force a mic string/permission and Data Safety entry — video capture records without audio and no mic permission is requested at runtime.
- [ ] Verify EXIF/GPS is stripped before upload and reflect that in permission/privacy disclosures (**security**) — GPS metadata leakage from photos is a privacy exposure; EXIF/GPS is stripped pre-upload — an uploaded image on device carries no GPS EXIF, and location handling is disclosed consistently with the coarse permission.

## Health Declarations

- [ ] Declare the app as informational triage, NOT diagnosis/treatment, in both stores' health/medical declarations (**PRD-02 / ai-safety**) — a false negative is the #1 risk and both stores police medical overclaims; triage output is EMERGENCY/MONITOR/LIKELY NORMAL, never a diagnosis — health-declaration answers and copy state the app provides guidance and does not diagnose, treat, or replace a veterinarian.
- [ ] Confirm the non-strippable medical disclaimer ships on every result surface (**LEG portal / disclaimer**) — the server forces `disclaimer_required=True` (`pipeline.py`), parsed in Dart and UI-gated on both result screens; a health app must always show it — `verify-disclaimers.sh` is green and the disclaimer renders on EMERGENCY and non-EMERGENCY results on device.
- [ ] State the emergency behavior honestly: EMERGENCY guidance is shown free and directs the user to a vet, and offline emergency triage is limited (**QA-06**) — the keyword emergency override is server-only, so offline capture cannot run it and screens show no offline warning — declarations/first-run copy note that emergency detection requires connectivity and always advise contacting a vet directly in a real emergency.
- [ ] Do not claim clinical accuracy or model reliability without live-model monitoring (**AI-02**) — there is no live quality/regression monitoring, so a silently-regressing model (false NORMALs) is invisible; claiming accuracy would be unsupported — health declarations avoid quantitative accuracy claims until regression monitoring exists.

## Data Safety

- [ ] Rebuild the Google Play Data Safety mapping to match the SDKs actually bundled (**PLAY-02, HIGH, blocks Google**) — the in-repo mapping materially omits location, purchase history, device/analytics, and third-party AI sharing that the bundled SDKs (RevenueCat, PostHog, Sentry, OneSignal, Gemini/Claude image sharing) actually process — every data type collected/shared by a bundled SDK appears in the Data Safety form with correct purpose and sharing flags.
- [ ] Declare image/video upload and third-party AI processing as data collection + sharing (**ai-safety / backend**) — photo/video pixels are sent to Gemini and Claude for triage (real multimodal, per **AI-01**); this is "sharing with third parties" for Data Safety — the form declares photos/videos are collected and shared with AI processors, with the triage purpose.
- [ ] Declare coarse location, purchase history, and analytics/crash data (**PLAY-03 / monetization / analytics**) — location (vet-finder), RevenueCat purchase history, and PostHog/Sentry analytics-crash data are all processed but under-declared — each is listed with accurate collection/sharing and linked-to-user status.
- [ ] Mirror the same disclosures in Apple's App Privacy "nutrition label" (**store-apple**) — Apple requires an equivalent privacy label; an inconsistent Play/Apple pair invites rejection — the App Privacy label matches the Play Data Safety mapping data-type-for-data-type.
- [ ] Reflect that account deletion is available in-app and via web in the Data Safety deletion answers (**5.1.1(v) / PLAY-04**) — both stores ask whether users can request deletion; in-app deletion is real (`account_screen.dart:190 → DeleteAccountScreen`, `delete-account` Edge Function) but the web path is currently undeliverable — Data Safety answers "yes" for both in-app and web deletion, and both paths actually work.
